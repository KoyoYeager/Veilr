# Veilr 開発日誌 — 経験・方針・試行錯誤の記録

## 1. プロジェクトの始まり

### 1.1 初期状態

ユーザーが「一時的に作成したが本当にできているかわからない」という状態でプロジェクトを受け取った。READMEとソースコードは存在したが、ビルドが通らない状態だった。

**最初の問題:**
- NuGetパッケージ `Hardcodet.NotifyIcon.Wpf` のバージョン4.1.1は存在しなかった（最新は2.0.1）
- `ImplicitUsings` / `Nullable` が未設定で `List<>`, `IDisposable` 等のコンパイルエラー
- 実装は約40-50%完了。骨格はあるが消去モード・設定UI・エクスポートが未実装

### 1.2 方針決定

- まずビルドを通す（最小限の修正）
- 次に起動して動作確認
- 未実装機能を仕様書に沿って実装
- 実際に起動して画面スクリーンショットで検証

- csproj に `ImplicitUsings` / `Nullable` 追加
- → ビルド成功（警告0、エラー0）

### 2.2 未実装機能の一括実装

一度に全ての未実装機能を実装：
- 消去モードの処理ループ（タイマーベース）
- 画像エクスポート（SaveFileDialog）
- 設定UI（4タブ全て）
- SettingsViewModelバインディング

**教訓:** 一括実装は効率的だが、問題の切り分けが難しくなる。段階的に実装してテストする方が安全。

### 2.3 テスト方法の確立

デスクトップアプリのため、ブラウザMCPは使えない。以下の方法を確立：
- PowerShellで `CopyFromScreen` → スクリーンショット取得
- Win32 API (`EnumWindows`, `GetWindowRect`) でウィンドウ位置検出
- `SetCursorPos` + `mouse_event` でボタンクリックシミュレーション
- `SendKeys` でキーボードショートカットテスト

**教訓:** WPFデスクトップアプリのUI自動テストはピクセル座標に依存するため脆い。ウィンドウ位置が変わるたびに座標再計算が必要。

## 3. UI設計の反復

### 3.1 シートの色統一問題

**ユーザーの指摘:** ドラッグバー・シート本体・ツールバーの間に明暗差がある。

**原因:** 各エリアに異なるOpacity（+0.1, 基本, +0.15）を設定していた。

**修正:** 全体を1つの`Border`にまとめ、同一Opacity。

**教訓:** デザイン仕様書の「ドラッグバーは10%濃く」を忠実に実装したが、ユーザーは統一感を求めていた。仕様書よりユーザーのフィードバックを優先すべき。

### 3.2 ✕ボタンの位置

**ユーザーの指摘:** ✕ボタンをツールバーではなく右上に。一般的なウィンドウの配置。

**教訓:** 独自UIでも、ユーザーが慣れている一般的なパターンを尊重する。

### 3.3 文字色の自動選定

**問題:** シート色が暗い場合に白文字、明るい場合に黒文字が必要。

**実装:** `輝度 = 0.299R + 0.587G + 0.114B`、140超なら黒、以下なら白。

### 3.4 設定変更時の再起動

**ユーザーの指摘:** 言語だけでなく、色や透明度の変更も再起動が必要。

**修正:** `HasChanges` プロパティで全設定の変更を検出し、適用時に自動再起動。

**教訓:** 「設定の反映」はWPFのバインディングだけでは不十分な場合がある。再起動が最も確実。

## 4. シートモードの進化

### 4.1 半透明→乗算ブレンド

**最大の転換点の1つ。**

**初期実装:** 単純なアルファ半透明の赤いウィンドウを重ねる。

**ユーザーの指摘:** 「赤シートがうまく機能させられない。透明にしすぎると文字が読めてしまう。」

**調査結果:** 実物の赤シートは半透明ではなく**カラーフィルター（乗算ブレンド）**。赤い光だけを通し、緑と青を吸収する。

```
赤シート(255,0,0)の場合:
赤文字(255,0,0) → 結果(255,0,0) = 背景と同色 → 文字が消える
黒文字(0,0,0)   → 結果(0,0,0)   = 黒のまま → 読める
```

**修正:** シートモードをリアルタイムキャプチャ＋乗算ブレンドに変更。

**教訓:** 物理現象をデジタルで再現するとき、安易な近似（半透明）ではなく、原理を正確に再現する（乗算ブレンド）方が正しい結果を得られる。

### 4.2 点滅問題

**問題:** 200msごとにウィンドウを非表示→キャプチャ→表示を繰り返すため、ちかちか点滅。

**修正:** タイマーループを廃止し、イベント駆動のワンショットキャプチャに変更。
- ドラッグ移動完了時
- リサイズ完了時（300msデバウンス）
- F5/スペースで手動リフレッシュ

**教訓:** 実物の赤シートは「置いたら固定」。デジタルでも同じUXを提供すべき。

## 5. 消去アルゴリズムの進化（最大の技術的挑戦）

### 5.1 HSV時代

**アプローチ:** HSV色空間で色相(H)を比較。

**致命的問題:** 黒(H=0)と赤(H=0)の色相が同じ。許容範囲を上げると黒文字が消える。

**試行錯誤:**
- 無彩色フィルター（S<20, V<30で除外）→ 黒は保護されるが限定的
- 閾値の手動調整 → 赤と茶色の分離が困難

**教訓:** HSVは人間の知覚に合っていない。特に無彩色と有彩色の分離が根本的に弱い。

### 5.2 CIE Lab移行

**転換:** HSVからCIE Lab色空間に全面移行。

**利点:**
- 黒(L=0,a=0,b=0)と赤(L=53,a=80,b=67)のΔE≈117 → 完全に分離
- 明度(L)と色味(a,b)が独立 → 明暗に関係なく「赤」を特定可能

**実装:**
- RGB→XYZ→Lab変換（sRGBガンマ補正 + D65白色点）
- 色相距離（chrominance distance）= √((a₁-a₂)²+(b₁-b₂)²)

### 5.3 二値マスク→クロマキー（ユーザー提案による大転換）

**ユーザーの質問:** 「クロマキー合成はなぜきれいに色を抜くことができるの？」

**この質問が最大のブレークスルーをもたらした。**

**それまでの限界:** 二値マスク（消す/消さない）→ ハードエッジ → 膨張・平均化で対処 → パッチの積み重ね

**クロマキーの核心:** 二値ではなく連続的なアルファ値（0.0〜1.0）でブレンド。

```
従来: pixel → mark(yes/no) → replace
新:   pixel → alpha(0.0-1.0) → blend(original * α + background * (1-α))
```

**なぜ自分で辿り着けなかったか:**
1. 「色検出→置換」（画像処理）として問題をフレーミング
2. 「色分離→合成」（映像制作）として考えるべきだった
3. 二値マスクという前提を疑わなかった
4. 検索クエリが画像処理用語に偏っていた

**最大の教訓:** 問題を解く前に、同じ問題を既に解いている分野を探す。クロマキーは1960年代から50年以上の歴史がある確立技術。

### 5.4 色ファミリー自動拡張

**発見の経緯:** ループ2でYCbCrアルゴリズムをテストしたとき、ユーザーが別ページに移動。ターゲットRGB(194,70,48)と実際の赤文字RGB(230,1,0)のLab距離が49で、どの閾値でも検出不可。

**根本原因:** アルゴリズムの問題ではなく、ターゲット色指定の問題。

**解決:** Lab色相角度ベースの色ファミリーマッチ。ターゲットと同じ色相方向（±10°）のピクセルを自動検出。

```
ターゲット(194,70,48): Lab角度39.0°
純赤(230,1,0):         Lab角度40.0° → 差1.0° → 検出 ✅
例題ヘッダー(235,101,0): Lab角度54.6° → 差15.6° → 除外 ✅
```

**教訓:** ユーザーが色を1つ選ぶだけで全ての同系色に対応すべき。ピクセル単位の正確な色指定を要求してはいけない。

### 5.5 置換品質の改善

| 試行 | 方法 | 効果 |
|---|---|---|
| 単一最近傍 | 最も近い非マーク色で置換 | ピンクアーティファクト |
| 平均（6→12サンプル） | 複数近傍の平均色 | ムラ減少 |
| エッジスキップ（1px→2px） | マーク済み隣接をスキップ | 背景色に近づく |
| 中央値（16サンプル） | 外れ値に強い | さらに安定 |
| デスピル | エッジの色カスト除去 | 色にじみ除去 |
| 段階的デスピル | 距離に応じた強度 | 自然な遷移 |

### 5.6 定量的改善ループ

15ループの反復改善で残存赤ピクセルを測定：

| アルゴリズム | 初期 | 最終 | 改善率 |
|---|---|---|---|
| ChromaKey | 211 | 89 | -58% |
| LabMask | 345 | 345 | 0%（ユーザー要望で戻し） |
| YCbCr | 236 | 195 | -17% |

**教訓:** 定量的指標（残存ピクセル数）による改善は方向性が明確だが、ユーザーの主観的品質判断と必ずしも一致しない。ユーザーはLabMaskの元パラメータを好んだ。

## 6. カラーピッカーの苦闘

### 6.1 3つの失敗

1. **Mouse.Capture方式:** ウィンドウ非表示時にCaptureが効かない → クリック不能
2. **透明オーバーレイ方式:** `Background=Transparent` だとWPFのhit testが透過 → クリック不能
3. **座標ずれ:** DPI変換漏れでマウス位置とプレビュー位置がずれる

### 6.2 最終解決

**全画面スナップショット方式:**
- 画面全体をキャプチャして不透明なフルスクリーンウィンドウに表示
- クリックが確実に動作（不透明なので）
- GetCursorPosの物理座標をそのまま使用（DPI問題なし）
- 21x21ピクセルのズームプレビュー＋十字マーカー

**教訓:** 複雑なWin32/WPFの組み合わせよりも、シンプルな「画面全体をキャプチャして表示」の方が堅牢。

### 6.3 精度問題

**ユーザーの指摘:** スポイトで正確に位置を特定しても、ずれた色が取得される。

**原因:** WPFの`e.GetPosition()`（DIP座標）とスナップショット（物理ピクセル座標）のずれ。

**修正:** クリック時にWPFイベントではなく、タイマーで毎フレーム保存した`GetCursorPos`の物理ピクセル座標を使用。

## 7. アライメント問題

### 7.1 消去モードの画像ずれ

**ユーザーの指摘:** 「消去モードの画像がおかしい。全くもとの描画と違ってずれている。」

**原因（3つの複合）:**
1. `Stretch="UniformToFill"` がアスペクト比の違いで画像をクロップ
2. WPFのDIP座標と物理ピクセル座標のミスマッチ
3. `BorderThickness=1` がコンテンツ領域を1px縮める

**修正:**
- `Stretch="Fill"` に変更
- `GetWindowRect`（Win32）で物理ピクセル座標を直接取得
- BorderThickness除去
- 全ウィンドウキャプチャに統一（バーと本体の境界のつなぎ目を解消）

**教訓:** WPFの座標系とWin32の座標系の混在は危険。1つの座標系（物理ピクセル）に統一すべき。

## 8. 言語対応

### 8.1 resx未統合問題

**ユーザーの指摘:** 「言語を変更しても全く英語にならない。」

**原因:** resxファイルは存在したが、UIに接続されていなかった。全てハードコード日本語。

**修正:** `Loc`ヘルパークラスで全UI文字列を一元管理。起動時に`Loc.SetLanguage()`で設定。

**教訓:** resxの仕組みは複雑でセットアップに失敗しやすい。シンプルな静的クラスの方が確実。

## 9. 得られた原則

### 技術選定

1. **HSV < CIE Lab < クロマキー** — 色空間と手法は段階的に良くなった
2. **問題を解く前に隣接分野を探す** — 映像制作(クロマキー)は50年の歴史
3. **二値よりも連続値** — マスク(0/1)よりアルファ(0.0-1.0)が自然
4. **物理現象を正確に再現** — 半透明ではなく乗算ブレンド

### 開発プロセス

5. **定量指標で改善を測る** — 残存ピクセル数でアルゴリズムを比較
6. **ユーザーの主観 > 定量指標** — 数値が良くてもユーザーが好まないことがある
7. **パッチの積み重ねより根本解決** — 膨張→平均→スキップ...よりクロマキー1つ
8. **DIP/物理ピクセルの混在は事故の元** — 1つの座標系に統一

### ユーザーインタラクション

9. **仕様書よりユーザーのフィードバック** — ドラッグバー10%濃く → 統一が正解
10. **一般的なUIパターンを尊重** — ✕ボタンは右上
11. **ユーザーの質問が最大のヒント** — 「クロマキーはなぜ綺麗？」→ 技術的ブレークスルー
12. **色指定は1つでよい** — ユーザーに正確なRGB指定を要求しない
